# 01_feature_design.md: F03 - Clinic Billing & Invoicing

This document describes the technical architecture and specifications for the **Clinic Billing & Invoicing** features in the Spring PetClinic application.

---

## 1. Feature Overview
The Clinic Billing & Invoicing feature enables the clinic to charge pet owners for services (appointments, visits, tests, and prescriptions) and tracks payment status.

### Core Flows:
1. **Invoice Generation (Staff Flow)**:
   - Receptionists or vets can generate invoices. An invoice must be associated with an [Owner](file:///d:/SatishAIProjects/12-GitHub_NonSalesforce/src/main/java/org/springframework/samples/petclinic/model/Owner.java). It can optionally reference a [Visit](file:///d:/SatishAIProjects/12-GitHub_NonSalesforce/src/main/java/org/springframework/samples/petclinic/model/Visit.java) or [Appointment](file:///d:/SatishAIProjects/12-GitHub_NonSalesforce/src/main/java/org/springframework/samples/petclinic/model/Appointment.java).
   - Staff specify the amount, due date, and line item description.
2. **Invoicing Portal (Owner Flow)**:
   - Pet owners can view a list of invoices associated with their account.
   - Owners can view individual invoice details and click "Pay Now" to open a mock payment gateway.
   - Once payment is successful, the invoice status changes to `PAID` and `paymentDate` is set.

## 2. Database Schema (DDL)
We will introduce a new table `invoices` to represent billing records.

### New Table: `invoices`
```sql
CREATE TABLE invoices (
    id             INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    owner_id       INTEGER NOT NULL,
    visit_id       INTEGER,
    appointment_id INTEGER,
    amount         DECIMAL(10, 2) NOT NULL,
    issue_date     DATE NOT NULL,
    due_date       DATE NOT NULL,
    payment_status VARCHAR(20) DEFAULT 'UNPAID' NOT NULL, -- UNPAID, PAID, VOIDED, OVERDUE
    payment_date   DATE,
    description    VARCHAR(255) NOT NULL,
    FOREIGN KEY (owner_id) REFERENCES owners(id),
    FOREIGN KEY (visit_id) REFERENCES visits(id),
    FOREIGN KEY (appointment_id) REFERENCES appointments(id)
);
CREATE INDEX invoices_owner_id ON invoices (owner_id);
```

## 3. Backend Architecture & API Mappings

### Entity Mapping (Java)
We will introduce a new JPA entity `Invoice` mapping to the `invoices` table.

```java
package org.springframework.samples.petclinic.model;

import java.math.BigDecimal;
import java.time.LocalDate;
import jakarta.persistence.*;
import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.NotEmpty;

@Entity
@Table(name = "invoices")
public class Invoice extends BaseEntity {

    @ManyToOne
    @JoinColumn(name = "owner_id")
    @NotNull
    private Owner owner;

    @ManyToOne
    @JoinColumn(name = "visit_id")
    private Visit visit;

    @ManyToOne
    @JoinColumn(name = "appointment_id")
    private Appointment appointment;

    @Column(name = "amount")
    @NotNull
    private BigDecimal amount;

    @Column(name = "issue_date")
    @NotNull
    private LocalDate issueDate;

    @Column(name = "due_date")
    @NotNull
    private LocalDate dueDate;

    @Column(name = "payment_status")
    @NotEmpty
    private String paymentStatus = "UNPAID";

    @Column(name = "payment_date")
    private LocalDate paymentDate;

    @Column(name = "description")
    @NotEmpty
    private String description;

    // Getters and Setters...
}
```

### Repository Layer
Create `InvoiceRepository` interface.
```java
package org.springframework.samples.petclinic.repository;

import java.util.List;
import org.springframework.dao.DataAccessException;
import org.springframework.samples.petclinic.model.Invoice;

public interface InvoiceRepository {
    void save(Invoice invoice) throws DataAccessException;
    List<Invoice> findByOwnerId(Integer ownerId) throws DataAccessException;
    List<Invoice> findAll() throws DataAccessException;
    Invoice findById(int id) throws DataAccessException;
}
```

### Controller Layer
We will introduce `InvoiceController` with the following HTTP mapping endpoints:
- `GET /owners/{ownerId}/invoices`: Lists invoices for an owner.
- `GET /owners/{ownerId}/invoices/{invoiceId}`: Displays individual invoice details.
- `GET /owners/{ownerId}/invoices/{invoiceId}/pay`: Displays mock payment checkout form.
- `POST /owners/{ownerId}/invoices/{invoiceId}/pay`: Processes simulated credit card payment.
- `GET /admin/invoices`: Displays receptionist dashboard of all system invoices.
- `GET /admin/invoices/new`: Displays invoice creation form.
- `POST /admin/invoices/new`: Submits and saves a new invoice.

## 4. Frontend View Design
We will add JSP templates under `src/main/webapp/WEB-INF/jsp/`:
1. `invoices/invoiceList.jsp`: Lists invoices, showing issue dates, amounts, and statuses with colored badges.
2. `invoices/invoiceDetails.jsp`: Shows invoice details, line items, and a "Pay Now" button if unpaid.
3. `invoices/paymentForm.jsp`: Mock credit card form with card number, expiration, and CVV fields.
4. `invoices/invoiceForm.jsp`: Staff view to create new invoices, featuring dynamic dropdowns for owners and their visits/appointments.

---

## 5. Architectural & Security Constraints
* **SQL Injection Prevention**: Ensure all database retrieval methods use Spring Data JPA query annotations or standard JPA parameterized queries.
* **XSS Prevention**: Clean and sanitize descriptions. All JSP views will use JSTL `<c:out>` or standard Spring bind tags.
* **Double Payment Prevention**: The payment execution logic in the controller/service must check if the invoice is already paid to prevent double charging.
* **Spring Transaction Manager**: Mark service-level payment methods with `@Transactional` to ensure updates to the database are consistent.
